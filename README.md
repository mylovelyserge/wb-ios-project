# 🍔 Eat & Pay — учебное приложение доставки еды

iOS-приложение доставки еды, которое я пишу с нуля на SwiftUI, осваивая iOS-разработку. Проект строится последовательно, этап за этапом: от каталога товаров до корзины и собственной дизайн-системы.

## ✨ Что уже умеет приложение

- **Каталог категорий** — сетка категорий с картинками, данные с сервера
- **Список товаров** — товары выбранной категории: фото, цена, вес, рейтинг
- **Экран деталей товара** — открывается как модальный sheet:
  - фото на всю ширину, цена, название, описание
  - рейтинг со звёздами и счётчиком отзывов (с правильным склонением: 1 отзыв / 2 отзыва / 5 отзывов)
  - индикатор «избранное» по данным сервера
  - кнопка «В корзину» с фирменным градиентом и подтверждением по нажатию
- **Вкладки** — TabView «Каталог» и «Корзина», каждая со своей независимой навигацией
- **Корзина**:
  - добавление товара из карточки каталога и с экрана деталей
  - один товар = одна позиция, повторное добавление увеличивает количество
  - кастомный контрол «− N +», удаление позиции при нуле
  - подсчёт итоговой суммы и количества товаров
  - пустое состояние и живой бейдж со счётчиком на вкладке
- **Дизайн-система** — цвета и типографика вынесены в отдельный Swift Package

## 🛠 Стек и технологии

| Область | Решение |
|---|---|
| UI | SwiftUI |
| Язык | Swift 6 |
| Архитектура | MV + Services (`@Observable`-сервисы без ViewModel) |
| Общее состояние | `@Observable` + Environment (корзина одна на всё приложение) |
| Дизайн-система | Локальный Swift Package `DesignSystem` |
| Сетевой слой | [swift-openapi-generator](https://github.com/apple/swift-openapi-generator) — клиент генерируется из `openapi.json` на этапе сборки |
| Транспорт | swift-openapi-urlsession |
| Авторизация | Bearer JWT через кастомный `ClientMiddleware` |
| Логирование | Собственный `LoggingMiddleware` (метод / путь / статус) |

## 🏗 Архитектура

Проект организован по слоям:

```
wb-ios-project/
├── Models/               # Доменные модели и моки
├── Services/             # Сервисы (@Observable) и middlewares
├── Views/                # Экраны и переиспользуемые компоненты
├── DesignSystem/         # Swift Package: DSColors, DSTypography
└── openapi.json          # Спека, из которой генерируется сетевой клиент
```

Ключевые принципы:

- **Своя доменная модель + маппинг из DTO** — UI не зависит от формата API
- **Single source of truth** — у каждого факта одно место жительства, производные значения вычисляются, а не хранятся
- **Экраны самодостаточны** — детальный экран принимает `id` и сам загружает данные, поэтому его можно открыть откуда угодно
- **`NavigationStack` внутри каждой вкладки** — вкладки сохраняют своё состояние независимо
- **Глупые компоненты** — переиспользуемые вью принимают данные и замыкания, не зная о сервисах

## 🚀 Как запустить

1. Клонируй репозиторий:
   ```bash
   git clone https://github.com/mylovelyserge/wb-ios-project.git
   ```
2. Создай файл `Secrets.swift` в корне проекта (он в `.gitignore` и не хранится в репозитории):
   ```swift
   enum Secrets {
       static let apiToken = "твой-JWT-токен"
   }
   ```
3. Открой проект в Xcode и запусти — сетевой клиент сгенерируется автоматически при первой сборке.

> **Требования:** Xcode с iOS 26 SDK, симулятор или устройство на iOS 26+.

## 📚 Чему я учусь в этом проекте

Проект учебный: каждое решение здесь принято осознанно и разобрано «почему так». Среди пройденного:

- **навигация** — `NavigationStack`, модальные экраны через `.sheet(item:)`, `TabView` и независимое состояние вкладок
- **состояние** — `@State`, `@Observable`-сервисы, общее состояние приложения через Environment
- **вёрстка** — стеки и выравнивание по базовой линии, `safeAreaInset`, `ContentUnavailableView`, тап-зоны и `contentShape`
- **данные** — генерация сетевого слоя из OpenAPI-спеки, маппинг DTO в доменные модели, работа с опционалами (chaining, nil-coalescing)
- **проектирование** — инварианты модели, вычисляемые свойства вместо хранимых, «глупые» переиспользуемые компоненты
- **модульность** — свой Swift Package, публичный API модуля и `public`-доступ
